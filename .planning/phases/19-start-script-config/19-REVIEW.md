## Summary

Reviewed `mesh_status/persistence.py` and `start.sh` at standard depth. Two blocker issues found: `_append_results` in `persistence.py` has a destructive overwrite bug that causes incremental data loss from on-disk daily files, and `persist_env` in `start.sh` fails to persist `LEADER_PORT` and `NODE_URL` to `.env`, causing configuration loss on restart. Several warnings include dead code (unused function, non-functional trap), missing input validation, and shell scripting robustness issues.

## Critical Issues

### CR-01: `_append_results` overwrites daily file with only the latest batch, causing data loss

**File:** `mesh_status/persistence.py:24-36`
**Issue:** `_append_results` opens the `.tmp` file in `"a"` (append) mode when the target `.json` exists, but this is equivalent to `"w"` (write) because the `.tmp` file is freshly created (it was renamed to the `.json` path via `os.replace` in the previous call). As a result, each call to `_append_results` creates a `.tmp` file containing ONLY the current batch of results, then replaces the real file with it — overwriting all previously-written entries.

For example:
- Flush cycle 1: writes day-1 entries (hours 0-1) to `day1.json`
- Flush cycle 2: writes day-1 entries (hours 0-2) to `day1.json` → overwrites, still correct (superset)
- ...continues until day-1 entries exceed 90h old...
- Flush cycle N (entries >90h old): calls `_flush_results` → writes day-1 entries to `day1.json` (correct, flush happens before move)
- `_move_old_to_aggregates` then removes day-1 entries from `_results`
- Flush cycle N+1: day-1 entries no longer in `_results` → `day1.json` is OVERWRITTEN without them → **LOST**

This means on-disk data older than ~90h is systematically purged from the daily JSONL files. On restart, `load_into_memory` cannot reconstruct `_day_aggregates` for those days. Historical data older than ~90h is permanently lost after any process restart or for the 90d dashboard after one extra flush cycle.

**Fix:** `_append_results` must either (a) read existing content and merge with new entries before writing, or (b) append directly to the real file (sacrificing atomic writes per entry). Option (a) preserves atomicity:

```python
def _append_results(d: date, results: list[dict]):
    if not results:
        return
    path = _date_path(d)
    path.parent.mkdir(parents=True, exist_ok=True)
    tmp_path = path.with_suffix(".tmp")

    # Read existing entries to preserve them
    existing = []
    if path.exists():
        with open(path) as f:
            for line in f:
                line = line.strip()
                if line:
                    existing.append(line)

    with open(tmp_path, "w") as f:
        # Write existing entries first
        for line in existing:
            f.write(line + "\n")
        # Append new entries
        for item in results:
            f.write(json.dumps(item, default=str) + "\n")
        f.flush()
        os.fsync(f.fileno())
    os.replace(tmp_path, path)
```

### CR-02: `persist_env` fails to persist `LEADER_PORT` and `NODE_URL`, causing configuration loss on restart

**File:** `start.sh:121-129`
**Issue:** The `persist_env` function writes a hardcoded template that only saves `LEADER_URL`, `LEADER_HOST`, and `DATA_DIR`. Two critical variables are omitted:

1. **`LEADER_PORT`:** If a user interactively sets a custom port (line 104-106), it is stored in the shell variable `LEADER_PORT` but never written to `.env`. On restart, the port defaults back to 58080 and the leader binds to the wrong port, or in a containerized setup the configured port mapping breaks.

2. **`NODE_URL`:** If a user interactively configures a node URL (line 116-117) or passes it via `--node-url`, it is used during the current session but never persisted. On restart, the node cannot report its own URL to the leader.

This was previously identified in planning documents (`.planning/v0.8-INTEGRATION.md` line 306: "stores them in local shell variables ... NEVER written back to `.env`") but was not fixed.

**Fix:**

```bash
persist_env() {
    cat > "$INSTALL_DIR/.env" <<EOF
# mesh-status configuration
# Generated by start.sh on $(date)
LEADER_URL=$LEADER_URL
LEADER_HOST=${LEADER_HOST:-0.0.0.0}
LEADER_PORT=${LEADER_PORT:-58080}
DATA_DIR=${DATA_DIR:-$INSTALL_DIR/data}
NODE_URL=$NODE_URL
EOF
}
```

## Warnings

### WR-01: Redundant sort in `_read_results` wastes computation

**File:** `mesh_status/persistence.py:58,123`

`_read_results` sorts results in ascending order by timestamp (line 58). `load_into_memory` then calls `_read_results` and immediately re-sorts in descending order (line 123). The first sort is completely wasted — `O(n log n)` computation thrown away.

The sort on line 58 should be removed since the sole caller (`load_into_memory`) re-sorts. If `_read_results` has future callers that need sorting, make it accept an optional sort parameter instead of always sorting.

**Fix:**

```python
# Line 58: Remove the sort from _read_results
# (the sole caller in load_into_memory sorts on line 123)
```

### WR-02: Trap handler is dead code; never fires after `exec`

**File:** `start.sh:152`

The trap `trap 'rm -f "$PID_FILE"; exit' SIGTERM SIGINT` is set on line 152, but line 153 immediately executes `exec uv run hypercorn ...` which replaces the shell process. Shell signal traps are **not** inherited across `exec`. The trap handler never executes in practice.

The PID file cleanup fails on SIGTERM/SIGINT. While the stale PID file is handled at startup (the `kill -0` check on line 144 fails for a dead process, and `rm -f` on line 148 cleans it), the cleanup trap is misleading dead code.

**Fix:** Remove the trap or implement cleanup before exec via process groups or a wrapper:

```bash
# Option A: Remove the dead trap (since stale PID cleanup works on restart)
# Just delete line 152

# Option B: Use a sub-shell to keep trap alive
(
    echo $$ > "$PID_FILE"
    trap 'rm -f "$PID_FILE"; exit' SIGTERM SIGINT
    exec uv run hypercorn mesh_status.leader:app --bind "${LEADER_HOST:-0.0.0.0}:${LEADER_PORT:-58080}"
)
```

### WR-03: Unquoted `$NODE_ARGS` vulnerable to word-splitting

**File:** `start.sh:170-172`

Building `NODE_ARGS` as a plain string (`NODE_ARGS="$NODE_ARGS --leader-url $LEADER_URL"`), then expanding it unquoted (`exec uv run python node.py $NODE_ARGS`) causes word-splitting on the argument values. If `LEADER_URL` or `NODE_URL` contained spaces or shell metacharacters, the arguments would be malformed. While URLs rarely contain spaces, this is a fragile shell pattern.

**Fix:** Use a bash array instead of a flat string:

```bash
NODE_ARGS=()
[ -n "$LEADER_URL" ] && NODE_ARGS+=(--leader-url "$LEADER_URL")
[ -n "$NODE_URL" ] && NODE_ARGS+=(--node-url "$NODE_URL")
exec uv run python node.py "${NODE_ARGS[@]}"
```

### WR-04: `--leader-url` with missing second argument silently consumes next arg

**File:** `start.sh:45`

```bash
--leader-url) LEADER_URL="$2"; shift ;;
```

If the user runs `start.sh --leader-url` without providing a URL value, `$2` expands to the next CLI argument (e.g., `--node`), which is then consumed as the URL value. This silently corrupts subsequent option parsing.

**Fix:** Validate that `$2` exists and doesn't start with `--`:

```bash
--leader-url)
    if [ $# -lt 2 ] || [[ "$2" =~ ^-- ]]; then
        echo "Error: --leader-url requires a URL argument"
        exit 1
    fi
    LEADER_URL="$2"; shift ;;
```

(Apply the same pattern to `--node-url` on line 46.)

### WR-05: `persist_env` writes `LEADER_HOST` even for node configuration where it is irrelevant

**File:** `start.sh:121-129`

The same `.env` template is written for both leader and node roles. For a node, `LEADER_HOST` is never used by `node.py`, but `NODE_URL` is not persisted. The template should be role-specific to avoid writing irrelevant config values that could confuse users inspecting `.env`.

**Fix:** Use separate templates or conditionalize the output:

```bash
persist_env() {
    cat > "$INSTALL_DIR/.env" <<EOF
# mesh-status configuration
# Generated by start.sh on $(date)
EOF
    if [ "$ROLE" = "leader" ]; then
        cat >> "$INSTALL_DIR/.env" <<EOF
LEADER_URL=$LEADER_URL
LEADER_HOST=${LEADER_HOST:-0.0.0.0}
LEADER_PORT=${LEADER_PORT:-58080}
DATA_DIR=${DATA_DIR:-$INSTALL_DIR/data}
EOF
    elif [ "$ROLE" = "node" ]; then
        cat >> "$INSTALL_DIR/.env" <<EOF
LEADER_URL=$LEADER_URL
NODE_URL=$NODE_URL
EOF
    fi
}
```

### IN-02: Default `DATA_ROOT` is a relative path that may resolve incorrectly without `.env`

**File:** `mesh_status/persistence.py:11`, `start.sh:127`

```python
DATA_ROOT = Path(os.environ.get("DATA_DIR", "data"))
```

When `persist_env` writes `.env`, it sets `DATA_DIR` to an absolute path (`$INSTALL_DIR/data`). But if the script is run in non-interactive mode (CLI flags only, no `.env` generated), `DATA_DIR` is not set, and `persistence.py` defaults to `"data"` — a relative path resolved from wherever the process starts. This could write to an unexpected location.

### IN-03: `$LEADER_URL` expansion in heredoc overwrites parameter expansion intended for runtime

**File:** `start.sh:125-127`

The `persist_env` function uses an unquoted heredoc, so `${LEADER_URL}` is expanded at write time into the file. But `LEADER_HOST=${LEADER_HOST:-0.0.0.0}` also gets its expansion at write time, producing a literal like `LEADER_HOST=0.0.0.0` in the `.env`. When `.env` is sourced later, `LEADER_HOST=0.0.0.0` is just an assignment — the `:-` fallback is lost because it was already resolved. This means the `.env` file contains literal values, not expressions. This is correct in practice but makes the `.env` file less transparent — a user editing it expects to see `LEADER_HOST=somevalue` rather than `LEADER_HOST=${LEADER_HOST:-0.0.0.0}`.

The `DATA_DIR` line similarly expands `$INSTALL_DIR` at write time. This is also correct but may surprise someone reading the generated file.
