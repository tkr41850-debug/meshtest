### Remaining Tech Debt

- **PID trap invalidation**: `start.sh` uses `exec` which clears bash traps. The PID file is not cleaned on shutdown, but the stale-PID check on next startup handles it.
- **Local-source CI path**: CI tests use `MESH_STATUS_LOCAL_SOURCE` instead of cloning from a remote URL. Does not test the real git-clone flow.
- **No shellcheck in CI**: CI has `ruff format --check`, `ruff check`, and `pyright` but no `shellcheck` for the bash/POSIX scripts.
- **No Nyquist validation**: No VALIDATION.md files exist for v0.8 phases (Nyquist is disabled for this milestone).

## E2E Flow Verification

### Flow: Install → Start Leader → Health Check

```
User runs: curl -fsSL https://install.mesh-status.dev | sh
  → deploy/install.sh checks prereqs (uv, git, curl)
  → Clones repo / copies source to ~/.local/meshtest
  → uv sync installs Python dependencies
  → npm ci + npm run build builds frontend
  → .env generated with defaults
  → Success banner printed

User runs: ~/.local/meshtest/start.sh --leader
  → OR: interactive wizard prompts for role and settings
  → .env loaded with set -a (exports DATA_DIR, LEADER_URL, etc.)
  → PID file written to var/leader.pid
  → Stale PID check prevents duplicate start
  → exec uv run hypercorn on port 58080

User visits: http://localhost:58080
  → Leader serves dashboard with /livez endpoint
  → Data files written to $DATA_DIR (from .env or env var)
```

### Flow: Uninstall

```
User runs: ~/.local/meshtest/start.sh --uninstall
  → rm -rf $INSTALL_DIR
  → Instructions to remove PATH line from shell rc
```

### Flow: CI Gate

```
PR pushed to main
  → GitHub Actions: Build Dockerfile.ci-test
  → Test 1: install.sh -y non-interactive
  → Test 2: install.sh -y with MESH_STATUS_VERSION=v0.8
  → Test 3: start.sh --leader, curl /livez up to 30s
  → Lint: ruff format --check, ruff check, pyright
```
