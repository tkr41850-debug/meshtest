### BLOCKER 1: Node has no HTTP server вЂ” peer push and config push are dead-lettered

**Files:** `leader.py:179-191` (`_notify_node`), `node.py` (no server)
**Issue:** The leader attempts to push peer lists to nodes via `POST http://{node_ip}:58080/update-peers` after every registration. Node.py never starts an HTTP listener. All push notifications silently fail.
**Affected requirements:** REGI-03, NODE-07
**E2E impact:** After registering, nodes never receive the updated peer list. If a new node joins, existing nodes won't know about it until their next check cycle (when they poll `GET /node-list`). The push mechanism exists in the protocol but has no server-side implementation.
**Severity:** BLOCKER вЂ” the system will work (nodes poll `/node-list`), but the push optimization and config push features are non-functional.

### BLOCKER 2: No `/healthz` endpoint on the leader (or anywhere)

**Files:** `node.py:73-74`, `leader.py:40-47` (has `/livez` and `/readyz` but not `/healthz`)
**Issue:** Node agents perform `GET /healthz` against each peer's port 58080 during every check cycle. The leader only exposes `/livez` and `/readyz`. Nodes running `node.py` don't serve HTTP at all. Every HTTP health check will fail with a connection error or 404.
**Affected requirements:** NODE-03
**E2E impact:** HTTP health check status is always `NotAvailable`/failed, making the HTTP dimension of the mesh connectivity view always show failures.
**Severity:** BLOCKER вЂ” the HTTP half of dual-check (ping + HTTP) is broken.

### BLOCKER 3: 30m data endpoint does not read from disk (data loss after flush)

**Files:** `leader.py:125-139`, `persistence.py:68-83`
**Issue:** The `/data?window=30m` handler iterates only over in-memory `_results`. The hourly flush loop retains only the last 10 minutes in memory (`persistence.py:78-82`). After a flush cycle, at most 10 minutes of data are available. The 30m endpoint should also read from disk (like the 30d endpoint does) to backfill.
**Affected requirements:** DATA-04
**E2E impact:** The 30-minute dashboard view shows incomplete data for up to 50 minutes after each flush. Users see at most 10 minutes of data, not 30.
**Severity:** BLOCKER вЂ” the 30m data window contract is broken.

### WARNING 1: `/updateConfig` pushes peer list, not config

**Files:** `leader.py:202-209` (`_push_config_to_all`)
**Issue:** The config push function calls `_notify_node` which sends `{"peers": ..., "check_interval": ..., "buffer_size": ...}`. A config update triggers a full peer push alongside the config values. This is semantically incorrect вЂ” changing the check interval shouldn't re-announce the full peer list (and vice versa, the push uses the same function).
**Affected requirements:** NODE-07

### WARNING 2: Persistence tests write to production `data/` directory

**Files:** `tests/test_persistence.py`, `persistence.py:12` вЂ” `DATA_ROOT = Path("data")`
**Issue:** Test artifacts (`data/2026/06/18.json`) are created in the real working directory. Running tests in a directory that has a `data/` folder with real data could corrupt production files. Tests should use a temp directory.
**Affected requirements:** None directly вЂ” test hygiene issue.

### WARNING 3: No Streamlit в†” leader deployment automation

**Files:** `README.md` вЂ” manual steps only
**Issue:** The dashboard and leader run as independent processes. There's no docker-compose, supervisor config, or startup script to coordinate them. The deployer must manually start the leader (`python -m mesh_status`) and dashboard (`streamlit run mesh_status/dashboard.py`) in separate terminals.
**Affected requirements:** DASH-01
