# Phase 4 — Streamlit Dashboard UI Design Contract

> Visual and interaction contract for the Streamlit dashboard frontend. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | None (Streamlit native components only) |
| Component library | Streamlit built-in — `st.tabs`, `st.table`, `st.expander`, `st.columns`, `st.markdown`, `st.metric`, `st.warning`, `st.caption`, `st.spinner`, `st.stop` |
| Icon library | None — use text-only status indicators and emoji sparingly (⚠ for warnings only) |
| Font | Streamlit default (system-ui / -apple-system / sans-serif cascade) — no custom font loading |

## Component Specifications

### 1. Connectivity Matrix Table

The primary visual — a grid showing status per-source, per-target.

**Structure:**
- Built using `st.table()` or `st.dataframe()` with custom column configuration
- Row header: source node IP address (monospace, left-aligned)
- Column headers: target node IP addresses (monospace, center-aligned, truncated or wrapped as needed)
- Cell content: centered status badge

**Badge rendering within cells:**
- OK: filled circle "●" or rounded-rect badge in `#22c55e` with `#f0fdf4` background
- NotAvailable: filled circle "●" or rounded-rect badge in `#f59e0b` with `#fffbeb` background
- Pending: filled circle "●" or rounded-rect badge in `#9ca3af` with `#f9fafb` background
- Self (diagonal, node_ip === target_ip): centered text "—" in `#9ca3af`

**Sorting:**
- Rows sorted by node IP (ascending)
- Columns sorted by target IP (ascending)
- Diagonal always runs top-left to bottom-right

**Column behavior:**
- First column (source node): fixed width for IP display (~140px)
- Data columns: equal proportional width, min 48px wide
- Horizontal scroll if >10 nodes (Streamlet auto-scroll)

### 2. Status Badge

| Property | Value |
|----------|-------|
| Shape | Rounded rectangle (4px border-radius) or filled circle |
| Min size | 20×20px |
| Padding | 4px 8px (when showing label), 4px (when showing icon-only) |
| Font | 14px monospace for icon, 12px for label text |
| Accessibility | No color-only indication — include text label tooltip on hover |

**30-Minute View badge shows:** Combined status (ping + HTTP). If both are OK → OK; if any is NotAvailable → NotAvailable; if no data → Pending.

**30-Day View badge shows:** Uptime percentage as color-coded text:
- ≥ 99%: badge in `#22c55e`
- ≥ 95% and < 99%: badge in `#f59e0b`
- < 95%: badge in `#ef4444` (red-500, new color for degraded performance)
- No data: badge in `#9ca3af` with text "—"

### 3. Per-Peer Expander Details

For each source node row, a `st.expander` labeled with the source IP containing per-target detail rows.

**Expander header format:**
```
▶ {source_ip}  [{status_summary}]  —  {N} targets
```

Where `status_summary` is "All OK", "X of Y down", or "Pending" based on current data.

**Expanded content — detail rows per target:**
Each target rendered as a `st.columns([1, 2, 2, 2, 1])` layout:

| Column | Content | Example |
|--------|---------|---------|
| 1 | Status badge (same as matrix) | Colored dot |
| 2 | Target IP (monospace) | `10.0.0.2` |
| 3 | Ping latency | `12.3ms` or `—` |
| 4 | HTTP latency | `45.1ms` or `—` |
| 5 | Last check time | `14:23:11` |

**30-Day View expander shows:**
For each day, a row showing:
- Status badge (color-coded by uptime %)
- Date label
- Ping uptime % (numeric)
- HTTP uptime % (numeric)
- Total checks count

### 4. Warning Banner

When leader is unreachable (HTTP request fails or times out):

```
┌──────────────────────────────────────────────────────────┐
│ ⚠ Leader unreachable — showing cached data              │
└──────────────────────────────────────────────────────────┘
```

- Rendered with `st.warning()` using Streamlit default warning styling
- Always appears at the top of the page body (below title, above tabs)
- Remains visible until next successful data fetch
- Cached data remains visible underneath

### 5. Auto-Refresh Indicator

Below the tab content area, a muted status line:

```
🔄 Auto-refreshing every 30s  |  Last update: 14:23:41
```

- Text: `#9ca3af` (gray-400), 14px caption
- Updated each fragment refresh cycle
- Does NOT appear during loading state (replaced by spinner)

## Design System Alignment

| Aspect | Alignment |
|--------|-----------|
| Rendering pipeline | Pure `st.*` components — no custom HTML/CSS, no JavaScript, no iframes |
| State management | Streamlet `session_state` for tab index persistence across reruns |
| HTTP client | `requests` (synchronous) for data API calls — Streamlit runs on main thread |
| Data caching | `@st.cache_data(ttl=30)` for 30-minute view; separate cache key for 30-day view |
| Fragment refresh | Single `@st.fragment` wrapping both tabs; `st.rerun(scope="fragment")` on 30s interval |
| Theme customization | Streamlit default light theme — no `.streamlit/config.toml` created |
| Charting | None — all data presented as tables and text |
| External dependencies | No new PyPI packages beyond `streamlit`, `requests` (already in pyproject.toml) |

## Interaction Patterns

### 1. Page Load Flow

```
User opens http://host:58581
  → st.title("mesh-status")
  → st.spinner("Loading mesh data...")
    → requests.get(LEADER_URL + "/data?window=30m")
    → requests.get(LEADER_URL + "/node-list")
    → @st.cache_data caches responses (ttl=30)
  → Spinner clears
  → st.tabs rendered with cached data
  → st.fragment auto-refreshes every 30s
```

### 2. Tab Switching

- `st.tabs(["30-Minute View", "30-Day View"])` with session_state tracking
- Tab index persisted in `st.session_state["active_tab"]`
- Switching tabs triggers data fetch for the selected window (cache hit if within TTL)
- Fragment scope includes both tabs — both views rerender but only active tab is visible

### 3. Auto-Refresh (30s interval)

- `@st.fragment` wraps entire tab content area
- Inside fragment: `time.sleep(30)` then `st.rerun(scope="fragment")`
- Cached data (ttl=30) expires on each refresh cycle, triggering fresh fetch
- Fragment rerun does NOT re-render page title, page-level config, or tab stubs
- Auto-refresh indicator timestamp updated on each successful fetch

### 4. Leader Unreachable

```
requests.get() raises ConnectionError or timeout (>5s)
  → st.warning("⚠ Leader unreachable — showing cached data")
  → Keep existing st.cache_data results visible (cache persists through error)
  → On next 30s tick: retry fetch
  → If successful: clear warning, update data
```

### 5. No Data State

```
Data API returns empty checks/statuses array for given window
  → st.info or st.markdown with "No data available for this time window"
  → Matrix not rendered (empty state replaces it)
  → Expanders not rendered
```

### 6. Loading States

- Initial load: full-page `st.spinner("Loading mesh data...")`
- Fragment refresh: no spinner (fragment rerun is silent — data updates in place)
- First fragment refresh after page load: data hot in cache, near-instant

### 7. Error Recovery

```
Any data fetch failure:
  → Log error (stderr via print or logging)
  → Show warning banner
  → Preserve cached/current data on screen
  → Do NOT clear state
  → Auto-retry on next 30s cycle
```

## Responsive Behavior

| Aspect | Behavior |
|--------|----------|
| Width | Streamlit defaults — content auto-widths to browser viewport |
| Min width | 768px recommended for matrix readability |
| Matrix | Horizontal scrollbar appears if columns exceed available width |
| Expander details | `st.columns()` wraps to next row on narrow viewports |
| Text | No truncation — IPs and status values are short strings |
| Mobile | Not a primary target — dashboard intended for desktop monitoring |
| Print | Not supported — dynamic data with auto-refresh |
